# EC618 OTA contract: what is frozen, and what each slot may change

The EC618 dual-slot OTA updates **only the VM slots** (`.vm_a` / `.vm_b`). The
rest of the AP image — PLAT (`.text`, drivers, the modem glue), the bootloader,
the CP image, and the fixed RAM layout — is the **base image** and is *not*
rewritten by an OTA. A firmware that is OTA'd onto a device therefore inherits
that device's base image, and it can only work if it agrees with the base on a
small, fixed interface.

This document is the authoritative list of that interface — the **contract**.
Changing anything here means the old base image and the new VM image disagree,
which faults *invisibly* (a stray pointer into an unmapped address, a call into
the wrong PLAT function), exactly the failure modes the relocation design fights.
Such a change is only safe as a **coordinated base + VM reflash**, never as an
OTA.

For the *mechanism* (relocate-on-write / un-relocate-on-read, the SRL3 table, the
container-in-slot layout) see [ota-relocation-convergence.md](ota-relocation-convergence.md);
for the storage/marker plan see [ota-dual-slot-plan.md](ota-dual-slot-plan.md).
The contract lives in the selected base ELF and ID, the linker template,
the anchor partition record, and the SRL3 framing implemented by
`src/slot_reloc_ec618.*` and `tools/ec618/slot-reloc.toit`.

---

## 1. Flash layout (base-specific)

The base ELF and anchor record define the immutable base regions, anchor
location, and the active partition table. A slot is linked once at
`TOIT_VM_LINK_BASE` and relocated into the selected table's slot A or B.
SRL3 carries the link base, slot size, and exact base ID.

A slot can run only when:

- its SRL3 base ID equals the flashed base ID;
- its declared slot size matches the destination partition;
- the active and candidate partition geometry is identical.

Partition-size or layout changes are deliberately rejected by the initial OTA
path. The anchor representation remains transactional so a future
migration-aware update can commit image, console, and partition state together.

**Frozen for one base:** base code and symbol addresses, link base, reserved VM
RAM ranges, anchor location/format, and the slot geometry exposed by that base.
**Free per slot:** everything *inside* a compatible slot (see §4).

---

## 2. Direct VM↔PLAT links

Each slot links separately with `--just-symbols=<base.elf>`. Symbols exported by
the selected base resolve to their exact fixed addresses; compiler-runtime
helpers absent from the base are pulled from the slot toolchain and live inside
the slot.

In-slot functions may be inlined normally. A non-inlined VM→PLAT call is a
direct Thumb branch. `gen-slot-reloc.toit` records every escaping branch in
SRL3, and relocate-on-write adjusts its immediate by the slot displacement.
The slot-B byte-identity oracle verifies that the recorded ABS32 and branch
sites are complete.

`plat_keep.c` retains a generous base symbol surface for future slots. A slot
that needs an unexported PLAT symbol fails to link and requires a new base
release. The base-ID gate makes symbol-address compatibility exact rather than
an assumed ABI.

---

## 3. RAM layout of `.load_dram_shared` (partly frozen)

PLAT loads `.load_dram_shared` (the writable `.data`) from a fixed flash LMA into
RAM once at startup. It holds, in order:

```
Image$$LOAD_DRAM_SHARED$$Base
  [ PLAT/SDK .data ]                  <- base-owned, live before the VM boots
  __vm_data_start
  [ VM + mbedtls .data ]              <- the PER-SLOT data region (§5)
  __vm_data_end
  [ PLAT .init_array ]
```

- The **PLAT `.data`** prefix is base-owned. It is live before the VM starts (PLAT
  mutates it), so it is **never** carried per-slot or overwritten. Its RAM
  position is frozen.
- `__vm_data_start` / `__vm_data_end` bracket the **VM's** writable `.data`
  (grouped contiguously). Their `{RAM base, length}` is part of the contract: the
  device copies the active slot's `.data` image to `[__vm_data_start,
  __vm_data_end)` at boot (§5), so the base and the OTA'd VM must agree on this
  range.

`src/toit_data_reloc.c` (auto-generated by `tools/ec618/gen-data-reloc.toit`,
verified by `make ec618 --check`) lists the RAM addresses within this range that
hold VM-slot pointers (the interpreter `dispatch_table`, the `*_primitives_`
tables). `relocate_data_slot_pointers()` shifts them by the slot delta at boot.

**Frozen:** the RAM position of the PLAT `.data` prefix and the
`[__vm_data_start, __vm_data_end)` range location.
**Free per slot:** the *contents* of the VM `.data` range (see §5).

## 4. The SRL3 table + slot framing (format frozen, contents free)

Each slot is self-describing via a tail trailer. The **canonical** image (what
`firmware.map` returns and `FirmwareWriter` consumes — the OTA payload) is
table-first; the **physical** slot stores it tail-first:

```
canonical:  [ size:u32 ][ SRL3 table ][ VM body + extension ][ VM .data init ]
physical:   [ VM body + extension ][ VM .data init ][ free ][ SRL3 table ][ size:u32 ]
```

The SRL3 header carries `link_base`, `slot_size`, `body_size`, `data_size`,
the counts for ABS32, Thumb-branch, and sector-straddling branch entries, plus
the required base version and 16-byte fingerprint. Delta-varint streams follow.
`body_size` is the relocatable populated front (VM body + in-slot extension);
`data_size` is the verbatim VM `.data` init image that rides after it (§5). The
size word is the slot's **last** word, so the trailer is self-locating.

This **format** is shared by the host builder (`tools/ec618/slot-reloc.toit`,
`gen-slot-reloc.toit`), the device engine (`src/slot_reloc_ec618.*`), and the
envelope/extract tools (`tools/firmware.toit`). All readers/writers must agree on
it — but the *table contents* (offsets, sizes) are rebuilt per firmware and are
**free**.

---

## 5. The per-slot VM `.data` init image (the A≠B fix)

**Why it exists.** The VM's writable `.data` (the `dispatch_table`, the
`*_primitives_` tables, and any mutable VM globals) differs between firmware
builds. Before this fix it was loaded once from the base image's fixed LMA — slot
A's `.data` — so a *different* slot-B firmware booted with slot A's `.data` values
and faulted. (Same-firmware A==B worked because the values coincided; a real
changed-code A≠B OTA did not.) Shifting the slot *pointers* by a delta cannot fix
this: the *contents* (e.g. each build's dispatch offsets) genuinely differ.

**The fix.** Each firmware carries its **own** `.data` init image inside its slot:

1. `gen-slot-reloc.toit` reads `[__vm_data_start, __vm_data_end)` from the link,
   extracts those init bytes from the slot's LMA in `ap.bin`, writes them to
   `slot-data.bin`, and records `data_size` in the SRL3 table.
2. The envelope bundles `slot-data.bin` (AR entry `$ec618-vm-data.bin`);
   `tools/firmware.toit` places it verbatim right after the body+extension in the
   slot (offset `body_size`), and appends it to the canonical image.
3. On write, the OTA streams body+extension+`.data` into the slot; the relocation
   never touches the `.data` region (its offsets are all `< body_size`), so it
   lands verbatim.
4. At boot, `load_active_slot_vm_data()` (`src/toit_ec618.cc`) copies the **active
   slot's** `.data` image over `[__vm_data_start, __vm_data_end)` — overriding the
   base's slot-A copy — *before* `relocate_data_slot_pointers()` shifts the (still
   link-base) slot pointers. The carried image holds link-base pointers, so the
   existing boot-time pointer relocation applies unchanged.

The `.data` image is **verbatim and never relocated**: it holds only link-base
pointers (fixed up in RAM at boot) and slot-independent values, so it reads back
identically from any slot. `data_size` must equal `__vm_data_end -
__vm_data_start` (the device asserts this at boot and skips the copy on mismatch
rather than overrunning RAM).

> **Deferred cleanup.** The pointer fix-up is still done at *boot*
> (`relocate_data_slot_pointers` + `toit_data_reloc.c`). Now that each slot
> carries its own `.data`, it could instead be relocated at *flash time* (extend
> the SRL3 `abs32` set to cover the `.data` region), dropping the boot-time pass
> and the generated table. Not done yet.

---

## 6. Changing the base image safely

Anything in §1–§3 (base symbols, slot geometry, link base, and the
`.load_dram_shared` RAM layout including the `__vm_data` bracket) is the
base⇄VM ABI. Changing it requires a **coordinated base + VM reflash** of every
device — an OTA of the VM alone will inherit the *old* base and fault. The build
guards enforce the format/consistency side of this:

- `check-slot-refs.toit` — no fixed (non-relocated) section points into a slot.
- `gen-slot-reloc.toit --verify-slot-b` — relocating slot A → slot B is
  byte-identical to an independent slot-B link (no dropped relocation).
- `gen-data-reloc.toit --check` — `toit_data_reloc.c` still matches the `.data`
  slot-pointer set.
- `tests/ctest/ec618-slot-reloc-test.cc` — the SRL3 parse + relocation engine,
  including the verbatim `.data` region.

§4–§5 (the SRL3 table contents and the per-slot `.data` image) are rebuilt per
firmware and are free to change with every OTA.
